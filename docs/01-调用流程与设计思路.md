# 调用流程与设计思路

## 1. 目标

这套 SDK 解决的是一个很具体的问题：

- 用户上传一个文档
- 或者业务表里已经保存了一个文件 URL
- 后端自动识别文件类型
- 将文件内容转换成后续可编辑的 HTML 字符串

这里的“可编辑”不是还原原文件，而是转换成前端富文本编辑器、正文编辑区可以继续修改的内容。

## 2. 两条入口链路

### 2.1 直接上传文件

SDK 不再内置 HTTP 接口，这条链路通常由你的业务项目自己写 controller，再调用 SDK 的
`DocumentImportService#importFile(MultipartFile file)`。

调用链：

1. 前端通过 `multipart/form-data` 上传文件
2. 业务 Controller 接收 `MultipartFile`
3. 调用 `DocumentImportService#importFile(MultipartFile file)`
4. Service 根据扩展名判断文件类型
5. 找到匹配的 `FileContentConverter`
6. converter 解析文件内容
7. 返回 HTML 字符串

### 2.2 通过 URL 回填内容

SDK 不再内置 HTTP 接口，这条链路通常由业务 service/controller 调用
`DocumentImportService#resolveContent(String content, String contentFileUrl)`。

调用链：

1. 业务请求传入：
   - `content`
   - `contentFileUrl`
2. 如果 `content` 本身有值，直接返回
3. 如果 `content` 为空，且 `contentFileUrl` 有值：
4. 调用 `RemoteFileDownloadUtils.download(url)` 下载文件
5. 取到：
   - 文件名
   - contentType
   - 文件二进制内容
6. 再调用 `DocumentImportService#importFile(String fileName, byte[] content, String contentType)`
7. 最终返回 HTML

这个流程就是你主项目里 `content + contentFile` 那套逻辑的简化版。

## 3. 为什么要设计成“统一入口 + 多 converter”

因为不同文件类型的解析方式完全不同。

例如：

- `txt` 只需要按文本分段
- `docx` 要解析段落、run、表格
- `doc` 要走老版 Word 的二进制解析
- `pdf` 只能抽文本
- `xls/xlsx` 需要按 sheet/row/cell 读取

如果把所有逻辑都写在一个方法里，会出现几个问题：

- `if else` 很长
- 后续维护很痛苦
- 新增新文件类型时容易把旧逻辑改坏
- 测试不方便

所以这里采用：

1. 统一入口：
   `DocumentImportService`
2. 统一扩展接口：
   `FileContentConverter`
3. 每种文件一个 converter

这是典型的“策略模式”。

## 4. Service 层为什么要保留两个 importFile 方法

### 4.1 `importFile(MultipartFile file)`

这个方法面向 Controller。

适合：

- 前端直接上传文件
- 接口调试
- 最基础的文件导入

### 4.2 `importFile(String fileName, byte[] content, String contentType)`

这个方法面向业务内部调用。

适合：

- 已经从 URL 下载了文件
- 文件来自数据库/对象存储
- 文件内容已经在内存里

这就是你主项目改造时最有价值的地方：业务层不用再伪造上传对象，直接传文件名和字节数组就能解析。

## 5. 为什么返回 String 而不是复杂对象

因为你的真实诉求是：

- 最终拿到一段可以放进正文编辑器的内容

所以 SDK 里做了收口：

- 文档类：直接返回 `richTextContent`
- 表格类：把所有表格 HTML 拼成一个 `<div>...</div>`

这样业务层只关心：

```java
String html = documentImportService.importFile(...);
```

不用再关心内部是 `txt/doc/docx/pdf/xlsx` 哪一种。

## 6. 主项目中的对应思路

你主项目里的发布模块，本质上就是这条逻辑：

1. `createPublish/updatePublish/submitPublish`
2. 先判断 `content` 是否有值
3. 如果没有值，再看 `contentFile`
4. 根据 `contentFile` 找到真实文件
5. 下载/读取字节数组
6. 调用导入器转成 HTML
7. 回填到 `content`
8. 再落库

所以核心思想只有一句：

业务系统里不要直接操作“文档格式”，而是尽量统一收口成“HTML 字符串”。
