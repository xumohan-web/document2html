## 2. SDK 和主项目的一一对应关系

### 2.1 上传解析入口

SDK：

- 不内置上传接口
- 由接入方 controller 调用 `DocumentImportService#importFile`

主项目：

- `PublishController#importContent`

### 2.2 统一解析服务

SDK：

- `DocumentImportServiceImpl`

主项目：

- `PublishDocumentImportServiceImpl`

### 2.3 按 URL 自动回填正文

SDK：

- `DocumentImportServiceImpl#resolveContent`

主项目：

- `PublishServiceImpl#resolvePublishContent`

### 2.4 内容写入业务表

SDK：

- 没有真正连数据库，只返回字符串

主项目：

- `PublishServiceImpl#createPublish`
- `PublishServiceImpl#updatePublish`
- `PublishServiceImpl#submitPublish`

它们最终都会把解析后的 HTML 放进 `content`。

## 3. 各 converter 的职责边界

### 3.1 TxtFileContentConverter

职责：

- 读取纯文本
- 按空行分段
- 输出 `<p>` 段落

### 3.2 DocFileContentConverter

职责：

- 处理 `.doc`
- 主要提取文本
- 是弱化支持，不追求复杂排版

### 3.3 DocxFileContentConverter

职责：

- 处理 `.docx`
- 解析段落、run、表格
- 尽量保留基础文本结构

### 3.4 PdfFileContentConverter

职责：

- 处理 `.pdf`
- 只做文本抽取
- 不保证复杂版式还原

### 3.5 ExcelFileContentConverter

职责：

- 处理 `.xls/.xlsx`
- 读取 sheet/row/cell
- 输出 table HTML

## 4. 为什么说这套结构适合后续扩展

假设以后你要支持：

- `markdown`
- `pptx`
- 图片 OCR
- 富文本块结构 JSON

你基本只需要做两件事：

1. 新增一个 converter
2. 在 `FileTypeEnum` 里注册类型

Service 和业务侧 controller 基本不用大改。

这就是这套结构最核心的价值。
